    Currency Logic Notes

    Board-report ready

    How is the currency conversion logic structured?

    Rates live in one lookup object keyed by currency code, with USD as the base. Because the provided rates are quoted as local currency per 1 USD, each expense is converted by dividing the original amount by its currency rate.

    Why structure it this way?

    The app uses one conversion function for the summary, merchant ranking, expense table, filters, added rows, and what-if rate changes. That keeps the math consistent and makes the output easier for a finance user to trust.

    What changes if a 25th currency is added tomorrow?

    Only the rate table needs a new currency code and valid rate. Once added, the currency appears in the expense form and the same summary logic can process new rows without a separate code path.

    What if a rate is null or missing?

    A missing or invalid rate would otherwise create bad USD values and distort totals. The app guards against that by validating rates before conversion, rejecting unsupported form entries, and showing a clear missing-rate state instead of hiding the issue.
